
SMART LEGAL DOCUMENT MANAGER
Product Requirements Document (PRD)

1. PRODUCT OVERVIEW
The Smart Legal Document Manager is a system designed to help legal teams manage document versions with full traceability.

The system ensures:
- Document history is never overwritten
- Lawyers can easily identify what changed between versions
- Every modification is recorded with timestamp and user information

The system includes:
- FastAPI backend
- PostgreSQL database
- React dashboard
- CLI interface
- Background notification mechanism

Primary goals:
- Version integrity
- Clear change comparison
- Simple and maintainable architecture
- Demonstration of product engineering thinking

4. CORE FUNCTIONAL REQUIREMENTS

4.1 Create Document

Input:
- title
- content
- created_by

Behavior:
- Document record created
- Version 1 created automatically
- Timestamp stored
- Audit log entry created


4.2 Upload New Version

Input:
- document_id
- content
- modified_by

Rules:
- Existing content must never be overwritten
- New version created with incremented version number


4.3 Prevent Identical Versions

If uploaded content is identical to the latest version:
- No new version is created
- System returns: "No meaningful change detected"

Whitespace differences should be ignored.


4.4 Version Comparison

Users must be able to compare two versions.

Input:
- document_id
- version_a
- version_b

Implementation must use Python difflib.

Output should clearly show:
- Added lines
- Removed lines
- Modified lines


4.5 Smart Notification System

When a new version is uploaded:
- Compare content with previous version
- Calculate similarity using difflib.SequenceMatcher

If similarity < 0.98:
Trigger notification.

Notifications run using FastAPI BackgroundTasks so upload response returns immediately.


4.6 Update Document Metadata

Users can update the document title without creating a new version.

Input:
- document_id
- new_title

Behavior:
- Update title
- Record audit log
- Do not create new version


4.7 Delete Document

Deletion must be soft delete.

Implementation:
is_deleted = True

Document history must remain intact.


4.8 Delete Specific Version

Users may delete individual versions.

Rules:
- Deleting a version must not delete the document
- The final remaining version cannot be deleted


4.9 Document Timeline

Users can view chronological activity.

Example:
v1 created by Sarang at 10:00
v2 edited by Jayesh at 11:15
title updated by Sarang at 12:00
